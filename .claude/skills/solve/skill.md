# solve — 完全自動開発サイクル

課題の受付から実装・テスト・PR・マージまでを一気通貫で自動実行する。
各ステップでマルチLLMレビュー（Claude Opus + Codex CLI）による品質ゲートを設け、合格するまでループする。

> ⛔ **絶対ルール**: `⛔ MANDATORY` マーカーが付いたステップは、修正規模・難易度に関係なく**絶対にスキップしてはならない**。
> 「小規模だから不要」「自明だから省略」という判断は禁止。Codex レビューは外部視点の品質ゲートであり、自己判断で省略する意味がない。
> また、`codex-code-review` サブエージェントは **Codex CLI を実際に起動する** エージェントである。Claude による代替レビューは Codex レビューとしてカウントしない。

---

## 前提条件

このスキルを使用するには、以下のツールがインストールされている必要がある:

| ツール | インストール | 用途 |
|--------|-------------|------|
| **Codex CLI** | `npm i -g @openai/codex` | 外部LLMによるコードレビュー |
| **GitHub CLI** | `brew install gh` / [公式](https://cli.github.com/) | Issue・PR・マージ操作 |
| **Node.js** | `brew install node` | ビルド・テスト実行 |

### カスタマイズが必要な項目

スキルを自分のプロジェクトに導入する際、以下の項目をプロジェクトに合わせて設定すること:

1. **`.claude/rules/review.md`** — プロジェクト固有のレビューチェック項目を定義
2. **`.claude/rules/coding-rules.md`** — コーディングルール（任意）
3. **`.claude/rules/testing.md`** — テストルール（任意）
4. **PR レビュー監視** — Phase 6 のリポジトリ名・監視対象ユーザーを自分のものに変更

## エージェント組織体制（必須）

あなた（メインエージェント）は**統括リーダー**として振る舞う。
エージェントチームを組織化し、プロジェクトを進行させること。

### 鉄則: メインコンテキストを汚染しない

```
🧠 メインエージェント（あなた）
│  役割: 指揮・判断・統合・品質ゲート・ユーザーとのコミュニケーション
│  禁止: コーディング、ファイル読み込み大量実行、詳細調査
│
├── 🔍 調査エージェント（Explore サブエージェント）
│   └─ Phase 1: 関連ファイル調査、影響範囲分析
│
├── 📋 プランナーエージェント（Plan サブエージェント）
│   └─ Phase 2: 実装プラン策定、技術選択肢の整理
│
├── 💻 実装エージェント（general-purpose サブエージェント）
│   └─ Phase 3: コーディング、コミット
│   └─ 複数ファイルが独立なら並列で複数エージェント起動
│
├── 🔎 Claude Opus レビューエージェント（opus-code-review サブエージェント）
│   └─ Phase 2-4, 3-2: 反復レビューループ（トークン効率重視）
│
├── 🤖 Codex レビューエージェント（codex-code-review サブエージェント）
│   └─ Phase 2-4, 3-2: 外部LLM最終ゲート（1回のみ）
│
├── 🧪 テストエージェント（general-purpose サブエージェント）
│   └─ Phase 4: テスト作成・実行
│
└── 🚀 PR・マージエージェント（general-purpose サブエージェント）
    └─ Phase 5-7: PR作成、レビュー監視、マージ
```

### メインエージェントがやること
- 課題を分析して**どのサブエージェントに何を任せるか**を決める
- 各サブエージェントの結果を受け取り、**判断・統合**する
- ユーザーへの確認（AskUserQuestion）と報告
- Phase 間の遷移判断（合格/不合格のゲート管理）
- サブエージェントが返した結果の**要約のみ**をコンテキストに保持

### メインエージェントがやらないこと
- ❌ 自分でコードを書く（実装エージェントに委譲）
- ❌ 自分でファイルを大量に読む（調査エージェントに委譲）
- ❌ 自分で詳細なプランを書く（プランナーエージェントに委譲）
- ❌ 自分でテストを書く・実行する（テストエージェントに委譲）

### 並列化の原則
- 独立したタスクは**常に並列**でサブエージェントを起動する
- 例: 調査中に Issue テンプレート準備 / FE と BE の実装を同時進行
- 依存関係がある場合のみ順次実行

## 実行モード

### 通常モード（デフォルト）
判断ポイントで AskUserQuestion を使い、ユーザーに確認を取る。

### 全自動モード（`--auto` フラグ）
ユーザーが `--auto` を付けた場合、すべての確認ポイントを **⭐推奨** で自動判断し、
一切の中断なしにマージまで走り切る。例外処理（Codex 5回不合格、テストFAIL解消不能）のみ停止。

```
/solve --auto ダッシュボードのグラフが表示されない
```

---

## ユーザー確認ポイント（通常モード: AskUserQuestion 必須 / 全自動モード: スキップ）

以下のタイミングでは、通常モードなら **必ず AskUserQuestion ツールで** ユーザーに確認を取る。
全自動モード（`--auto`）ではすべて ⭐推奨 を自動選択して進む。

| タイミング | 確認内容 | 全自動時の挙動 |
|-----------|---------|--------------|
| **Phase 1 完了時** | Issue内容・影響ファイル・ブランチ名 | 自動で作成 |
| **Phase 2 プラン提示時** | 選択肢の選択 or GO/NOGO | ⭐推奨を自動採用 |
| **Phase 3 実装方針の迷い** | 仕様曖昧・複数解釈 | 最もシンプルなアプローチを選択 |
| **Phase 4 テスト範囲** | 最低限 vs 網羅的 | ⭐推奨範囲で実行 |
| **Codex 5回不合格** | スキップ or 修正 | **停止してユーザーに確認** |
| **テスト FAIL 解消不能** | 方針確認 | **停止してユーザーに確認** |

**通常モードの原則**: 迷ったら聞く。手戻りより30秒の確認の方が安い。

## ワークフロー全体像

```
課題受付 → 関連ファイル調査 → [ask: Issue内容確認] → Issue作成 → ブランチ作成
    → プラン策定 → [ask: プラン承認]
    → Claude Opus レビューループ → Codex 最終ゲート
    → 実装
    → Claude Opus diff レビューループ → Codex 最終ゲート
    → テスト実行 → [ask: テスト範囲確認] → (不足時: テスト作成サイクル)
    → PR作成 → レビュー監視（10分/30秒間隔）
    → (レビュー対応サイクル) → マージ → 完了
```

## Phase 1: 課題分析・準備

### 1-1. 関連ファイル調査（→ 調査エージェントに委譲）

**Explore サブエージェント**を起動し、課題の影響範囲を調査させる:

```
サブエージェントへの指示:
- Glob と Grep で関連コンポーネント、フック、サービス、型定義を特定
- DB マイグレーション、サーバーレス関数の関連箇所も確認
- 影響範囲を「直接変更」「間接影響」に分類して報告
```

メインエージェントは結果の**要約のみ**受け取る。

### 1-2. GitHub Issue 作成

```bash
gh issue create \
  --title "{課題タイトル}" \
  --body "$(cat <<'EOF'
## 課題
{課題の説明}

## 影響ファイル
{洗い出したファイル一覧}

## 受け入れ条件
{完了条件のチェックリスト}
EOF
)"
```

Issue番号を取得して以降のブランチ名・PR紐付けに使用する。

### 1-3. ユーザー確認（AskUserQuestion）

Issue を作成する**前に**、以下を AskUserQuestion で確認する:

```
以下の内容でGitHub Issueを作成し、ブランチを切ります。

📋 Issue タイトル: {タイトル}
📂 影響ファイル: {ファイル一覧}
🔀 ブランチ名: {type}/#{仮番号}-{description}
✅ 受け入れ条件: {チェックリスト}

この内容で進めてよいですか？修正があれば教えてください。
```

ユーザーの承認後に Issue を作成する。

### 1-4. ブランチ作成

```bash
git checkout main
git pull origin main
git checkout -b {type}/#{issue番号}-{short-description}
```

ブランチ命名規則:
- `feat/`, `fix/`, `refactor/`, `docs/`, `test/`, `chore/`

## Phase 2: プラン策定・承認

### 2-1. 実装プランの作成（→ プランナーエージェントに委譲）

**Plan サブエージェント**を起動し、以下を含むプランを作成させる:

1. **ゴール定義**: 何を達成するか
2. **影響ファイル一覧**: 変更するファイルとその理由
3. **実装ステップ**: 具体的な手順（番号付き）
4. **技術的判断**: アーキテクチャ選択とその根拠

メインエージェントはプランの**要約を受け取り、ユーザーに提示**する。

### 2-2. 選択肢がある場合のユーザー確認

複数のアプローチが可能な場合、各選択肢を提示する:

```
### 選択肢A: {アプローチ名} ⭐（推奨）
- メリット: ...
- デメリット: ...
- 工数目安: ...

### 選択肢B: {アプローチ名}
- メリット: ...
- デメリット: ...
- 工数目安: ...
```

**⭐（推奨）を必ず1つ付ける。**

### 2-3. ユーザー承認 + 自動化レベル確認（AskUserQuestion）

プラン（単一 or 複数選択肢）を提示したら、**必ず AskUserQuestion で承認を取る**。
同時に、以降の進行モードを確認する:

```
上記のプランで実装を進めてよいですか？

📋 選択肢がある場合: どのアプローチを採用しますか？
✏️ 修正があれば教えてください

🚀 承認後の進め方:
  1. **通常モード** — 要所で確認しながら進めます（デフォルト）
  2. **全自動モード** — このままマージまで一気に進めます（確認なし、⭐推奨を自動採用）
```

- ユーザーが「2」「全自動」「おまかせ」「任せた」等と答えたら → 全自動モードに切り替え
- それ以外 → 通常モードで継続
- `--auto` フラグ付きで起動した場合はこの質問自体をスキップ

承認が得られるまで次のステップには進まない。

### 2-4. プランレビュー（2段階ゲート）⛔ MANDATORY — DO NOT SKIP

> **設計思想**: Claude Opus で反復修正（トークン安い）→ Codex CLI で最終ゲート（トークン高いが1回）。
> Codex のトークンを節約しつつ、外部視点の品質ゲートを確保する。

#### ステップ1: Claude Opus レビューループ（合格するまで繰り返す）

```
【ループ処理】
1. opus-code-review サブエージェント（または model: opus の general-purpose）を起動
   - レビュー対象: 実装プランのテキスト
   - 観点: 技術的妥当性、リスク、見落とし、プロジェクト固有ルール
2. P0/P1 の指摘があれば → プランを修正 → 再度 Claude Opus でレビュー
3. P0/P1 指摘なし（Claude Opus 合格）→ ステップ2 へ
```

#### ステップ2: Codex CLI 最終レビュー（1回のみ）

```
1. codex-code-review サブエージェントを起動（Bash で codex exec を実際に実行）
   - レビュー対象: Claude Opus 合格済みの実装プラン
   - 観点: Claude とは異なる外部LLMの視点での品質チェック
2. P0/P1 の指摘があれば → 修正 → ステップ1 に戻る（Claude Opus ループから再開）
3. P0/P1 指摘なし（Codex 合格）→ Phase 3 へ進む
```

## Phase 3: 実装

### 3-1. コーディング（→ 実装エージェントに委譲）

**general-purpose サブエージェント**を起動し、実装を委譲する:

```
サブエージェントへの指示:
- プロジェクトのコーディングルールに厳密に従うこと
- 各ファイル変更後にコミット（アトミックコミット）
- 完了したら変更ファイル一覧と変更概要を報告すること
```

**並列化**: 変更対象が独立している場合、複数の実装エージェントを並列起動する。
例: コンポーネント変更とサーバーレス関数変更を同時進行

メインエージェントは実装エージェントの完了報告を受け取り、Phase 3-2 へ進む。

### 3-2. 実装レビュー（2段階ゲート）⛔ MANDATORY — DO NOT SKIP

> **設計思想**: Phase 2-4 と同様、Claude Opus で反復修正 → Codex CLI で最終ゲート。

#### ステップ1: Claude Opus diff レビューループ（合格するまで繰り返す）

**opus-code-review サブエージェント（または model: opus の general-purpose）**を起動し、diff ベースの実装レビューを委譲する。
TypeScript の型チェック通過だけで完了にせず、**変更差分（diff）を基準に**実装レビューを行わせる。

```
【ループ処理】
1. git diff main...HEAD で変更差分を取得
2. 以下の5観点で差分を精査する:

   ✅ 仕様適合性:
     - 変更が Phase 2 で承認されたプランどおりか
     - Issue の受け入れ条件をすべて満たしているか

   ✅ ロジックの健全性:
     - 条件分岐・ループ・データフローに破綻がないか
     - 既存機能への副作用（意図しない挙動変化）がないか

   ✅ 変更の純度:
     - 不要な変更（フォーマット変更、無関係なリファクタ）が混ざっていないか
     - 変更の各行が課題解決に必要なものか

   ✅ エラーハンドリング・境界条件:
     - null/undefined、空配列、空文字列のケースは考慮されているか
     - API エラー、ネットワークエラー時の処理は適切か
     - ユーザー入力のバリデーションに漏れはないか

   ✅ プロジェクト固有ルール:
     - .claude/rules/review.md に定義されたチェック項目に違反していないか

3. P0/P1 の指摘があれば → 修正 → 再度 Claude Opus でレビュー
4. P0/P1 指摘なし（Claude Opus 合格）→ ステップ2 へ
```

#### ステップ2: Codex CLI 最終レビュー（1回のみ）⛔ MANDATORY — DO NOT SKIP

```
1. git diff main...HEAD で変更差分を取得
2. codex-code-review サブエージェントを起動（Bash で codex exec を実際に実行）
   - レビュー対象: git diff の実コード（Claude Opus 合格済み）
   - チェック項目: .claude/rules/review.md に定義されたプロジェクト固有チェック項目
3. P0/P1 の指摘があれば → 修正 → ステップ1 に戻る（Claude Opus ループから再開）
4. P0/P1 指摘なし（Codex 合格）→ Phase 4 へ進む
```

## Phase 4: テスト

### 4-1. 既存テスト実行（→ テストエージェントに委譲）

**general-purpose サブエージェント**を起動し、テスト実行を委譲する:

```bash
# サブエージェントが実行するコマンド（プロジェクトに合わせて変更）
npm run test
npx tsc --noEmit
npm run lint
```

メインエージェントは PASS/FAIL の結果サマリーのみ受け取る。

### 4-2. テスト結果による分岐

**全テスト PASS → Phase 5 へ**

**テスト不足・新機能でテストなし → AskUserQuestion でテスト範囲を確認:**

```
テストが不足しています。以下の方針でテストを作成してよいですか？

📝 テスト対象ファイル: {一覧}
🎯 テスト範囲:
  - 最低限: 変更した関数の正常系・異常系のみ
  - 推奨: 変更した関数 + 影響を受ける既存機能 ⭐

どちらの範囲で進めますか？
```

ユーザーの回答後、テスト作成サイクルに入る:

```
1. テスト作成プランを策定
   - 対象ファイルごとのテスト方針
   - 正常系・境界値・異常系の3ケース以上
   - AAA パターン（Arrange-Act-Assert）

2. Claude Opus テストプランレビューループ
   - opus-code-review でテストプランをレビュー
   - P0/P1 指摘 → 修正 → 再レビュー
   - Claude Opus 合格 → Codex 最終レビュー

3. Codex テストプラン最終レビュー（1回のみ）
   - codex-code-review で最終チェック
   - P0/P1 指摘 → 修正 → Claude Opus ループに戻る
   - 合格 → テスト実装へ

4. テストファイル実装
   - プロジェクトのテストルールに準拠

5. Claude Opus テスト実装レビューループ + Codex 最終レビュー
   - Phase 3-2 と同じ2段階ゲート

6. テスト実行
   - npm run test -- {テストファイルパス}
   - 全 PASS → Phase 5 へ
   - FAIL → 修正 → 再実行
```

## Phase 5: PR 作成（→ PR・マージエージェントに委譲）

### 5-1. 最終チェック

**general-purpose サブエージェント**に最終チェックとPR作成を一括で委譲する:

```bash
# サブエージェントが実行するコマンド
npx tsc --noEmit       # 型チェック
npm run lint           # ESLint
npm run test           # 全テスト
```

全 PASS を確認してからPRを作成する。

### 5-2. PR 作成

```bash
git push -u origin {ブランチ名}

gh pr create \
  --title "{Conventional Commits形式のタイトル}" \
  --body "$(cat <<'EOF'
## 概要
{変更内容を1-3文で}

Closes #{issue番号}

## 変更種別
- [x] {該当する種別}

## 変更ファイル
{変更ファイル一覧}

## テスト
- [x] 型チェック通過
- [x] テスト通過
- [x] Claude Opus レビュー合格
- [x] Codex CLI レビュー合格

🤖 Generated with [Claude Code](https://claude.ai/code)
EOF
)"
```

## Phase 6: PR レビュー監視・対応

### 6-1. レビュー監視ループ

```
【監視ループ: 最大10分間、30秒間隔】

PR番号を取得後:

while (経過時間 < 10分):
    sleep 30秒

    reviews = gh api repos/{owner}/{repo}/pulls/{PR番号}/reviews
    comments = gh api repos/{owner}/{repo}/pulls/{PR番号}/comments

    if 新しいレビュー/コメントあり（bot以外）:
        → Phase 6-2 へ（レビュー対応）
        → 対応完了後、タイマーを0分からリセットして監視再開

監視終了（10分間レビューなし）→ Phase 6-3 へ
```

**監視対象**: 人間のレビュアーによるコメント
**無視**: CI/CD bot、デプロイ bot のコメント

### 6-2. レビュー対応サイクル

```
1. レビュー内容を確認
2. 修正プランを策定
3. Claude Opus レビューループ + Codex 最終レビュー（2段階ゲート）
4. 修正を実装
5. コミット & プッシュ
6. 監視タイマーを 0分 からリセットして監視再開
```

### 6-3. マージ

10分間新規レビューがなければマージ:

```bash
gh pr merge {PR番号} --squash --delete-branch
```

## Phase 7: 完了

1. マージ完了を確認
2. 実施内容のサマリーを報告:
   - 解決した課題
   - 変更ファイル一覧

## レビューの共通ルール

### 2段階レビューゲートの設計思想

```
Claude Opus ループ（安い・反復向き）
    ↓ P0/P1 なしで合格
Codex CLI 最終ゲート（高い・外部視点・1回のみ）
    ↓ P0/P1 なしで合格 → 次の Phase へ
    ↓ P0/P1 あり → Claude Opus ループに戻る
```

- **Claude Opus**: 修正→再レビューの反復ループに最適（トークンコスト低い）
- **Codex CLI**: 異なるLLMの視点で最終チェック（Claude が見逃す問題を発見できる）

### 共通設定

- **合格基準**: P0/P1 の指摘が 0 件
- **P2（軽微）**: 報告のみ、修正は任意
- **最大リトライ**: Codex 5回不合格でユーザーに判断を仰ぐ
- **レビュー観点**: `.claude/rules/review.md` に準拠

## 中断・例外処理

- **ユーザーが中断指示**: 現在のステップを完了してから停止。進捗を報告
- **Codex レビュー5回不合格**: ユーザーに指摘内容を提示し、スキップか修正か判断を仰ぐ
- **テスト FAIL が解消できない**: 原因分析を報告し、ユーザーに判断を仰ぐ
- **ビルドエラー**: エラー内容を分析し、自動修正を試みる。3回失敗でユーザーに報告
