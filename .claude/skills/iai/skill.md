# iai — 居合

> 居合：抜刀から納刀までを一つの流れで行う剣術。このスキルも、課題の受付からマージまでを一気通貫で実行する。

各ステップでマルチLLMレビュー（Claude Opus + Codex CLI）による品質ゲートを設け、合格するまでループする。

> ⛔ **絶対ルール**: `⛔ MANDATORY` マーカーが付いたステップは、修正規模・難易度に関係なく**絶対にスキップしてはならない**。
> 「小規模だから不要」「自明だから省略」という判断は禁止。Codex レビューは外部視点の品質ゲートであり、自己判断で省略する意味がない。
> また、`codex-code-review` サブエージェントは **Codex CLI を実際に起動する** エージェントである。Claude による代替レビューは Codex レビューとしてカウントしない。

---

## 前提条件

このスキルを使用するには、以下のツールがインストールされている必要がある:

| ツール | インストール | 用途 |
|--------|-------------|------|
| **Codex CLI** | `npm i -g @openai/codex` | 外部LLMによるコードレビュー |
| **GitHub CLI** | `brew install gh` / [公式](https://cli.github.com/) | Issue・PR・マージ操作 |
| **Node.js** | `brew install node` | ビルド・テスト実行 |

### カスタマイズが必要な項目

スキルを自分のプロジェクトに導入する際、以下の項目をプロジェクトに合わせて設定すること:

1. **`.claude/rules/review.md`** — プロジェクト固有のレビューチェック項目を定義
2. **`.claude/rules/coding-rules.md`** — コーディングルール（任意）
3. **`.claude/rules/testing.md`** — テストルール（任意）
4. **PR レビュー監視** — Phase 6 のリポジトリ名・監視対象ユーザーを自分のものに変更

## エージェント組織体制（必須）

あなた（メインエージェント）は**統括リーダー**として振る舞う。
エージェントチームを組織化し、プロジェクトを進行させること。

### 鉄則: メインコンテキストを汚染しない

```
🧠 メインエージェント（あなた）
│  役割: 指揮・判断・統合・品質ゲート・ユーザーとのコミュニケーション
│  禁止: コーディング、ファイル読み込み大量実行、詳細調査
│
├── 🔍 調査エージェント（Explore サブエージェント）
│   └─ Phase 1: 関連ファイル調査、影響範囲分析
│
├── 📋 プランナーエージェント（Plan サブエージェント）
│   └─ Phase 2: 実装プラン策定、技術選択肢の整理
│
├── 💻 実装エージェント（general-purpose サブエージェント）
│   └─ Phase 3: コーディング、コミット
│   └─ 複数ファイルが独立なら並列で複数エージェント起動
│
├── 🔎 Claude Opus レビューエージェント（opus-code-review サブエージェント）
│   └─ Phase 2-4, 3-2: 反復レビューループ（トークン効率重視）
│
├── 🤖 Codex レビューエージェント（codex-code-review サブエージェント）
│   └─ Phase 2-4, 3-2: 外部LLM最終ゲート（1回のみ）
│
├── 🧪 テストエージェント（general-purpose サブエージェント）
│   └─ Phase 4: テスト作成・実行
│
└── 🚀 PR・マージエージェント（general-purpose サブエージェント）
    └─ Phase 5-7: PR作成、レビュー監視、マージ
```

### メインエージェントがやること
- 課題を分析して**どのサブエージェントに何を任せるか**を決める
- 各サブエージェントの結果を受け取り、**判断・統合**する
- ユーザーへの確認（AskUserQuestion）と報告
- Phase 間の遷移判断（合格/不合格のゲート管理）
- サブエージェントが返した結果の**要約のみ**をコンテキストに保持

### メインエージェントがやらないこと
- ❌ 自分でコードを書く（実装エージェントに委譲）
- ❌ 自分でファイルを大量に読む（調査エージェントに委譲）
- ❌ 自分で詳細なプランを書く（プランナーエージェントに委譲）
- ❌ 自分でテストを書く・実行する（テストエージェントに委譲）

### 並列化の原則
- 独立したタスクは**常に並列**でサブエージェントを起動する
- 例: 調査中に Issue テンプレート準備 / FE と BE の実装を同時進行
- 依存関係がある場合のみ順次実行

## 実行モード

### 通常モード（デフォルト）
判断ポイントで AskUserQuestion を使い、ユーザーに確認を取る。

### 全自動モード（`--auto` フラグ）
ユーザーが `--auto` を付けた場合、すべての確認ポイントを **⭐推奨** で自動判断し、
一切の中断なしにマージまで走り切る。例外処理（Codex 5回不合格、テストFAIL解消不能）のみ停止。

```
/iai --auto ダッシュボードのグラフが表示されない
```

---

## ユーザー確認ポイント（通常モード: AskUserQuestion 必須 / 全自動モード: スキップ）

以下のタイミングでは、通常モードなら **必ず AskUserQuestion ツールで** ユーザーに確認を取る。
全自動モード（`--auto`）ではすべて ⭐推奨 を自動選択して進む。

| タイミング | 確認内容 | 全自動時の挙動 |
|-----------|---------|--------------|
| **Phase 1 完了時** | Issue内容・影響ファイル・ブランチ名 | 自動で作成 |
| **Phase 2 プラン提示時** | 選択肢の選択 or GO/NOGO | ⭐推奨を自動採用 |
| **Phase 3 実装方針の迷い** | 仕様曖昧・複数解釈 | 最もシンプルなアプローチを選択 |
| **Phase 4 テスト範囲** | 最低限 vs 網羅的 | ⭐推奨範囲で実行 |
| **Codex 5回不合格** | スキップ or 修正 | **停止してユーザーに確認** |
| **テスト FAIL 解消不能** | 方針確認 | **停止してユーザーに確認** |

**通常モードの原則**: 迷ったら聞く。手戻りより30秒の確認の方が安い。

## ワークフロー全体像

```
課題受付 → 症状トリアージ → 仮説駆動調査 + Git考古学（並列）
    → 根本原因の特定 → [ask: 調査結果・Issue内容確認] → Issue作成 → ブランチ作成
    → プラン策定 → [ask: プラン承認]
    → Claude Opus レビューループ → Codex 最終ゲート
    → 実装
    → Claude Opus diff レビューループ → Codex 最終ゲート
    → テスト実行 → [ask: テスト範囲確認] → (不足時: テスト作成サイクル)
    → PR作成 → レビュー監視（10分/30秒間隔）
    → (レビュー対応サイクル) → マージ → 完了
```

## Phase 1: 深層調査・原因特定（Senior Engineer Methodology）

> **設計思想**: ジュニアは「関連ファイルを探す」。シニアは「仮説を立て、最小コストで検証し、根本原因を特定する」。
> このフェーズでは、シニアエンジニアのデバッグメソドロジーをAIエージェントに移植する。
> **原則: 闇雲に探さない。すべての調査行動は仮説の検証でなければならない。**

### 1-1. 症状トリアージ（メインエージェントが実行）

課題を受け取ったら、まず**症状を分類**して調査戦略を決定する。推測ではなく、与えられた情報を文字通り読む。

#### 課題タイプの自動分類

| タイプ | 判定基準 | 調査戦略 |
|--------|----------|----------|
| **CRASH** | スタックトレース・エラーメッセージあり | → コールスタック逆引き + 直近変更の相関分析 |
| **WRONG_DATA** | 期待値と実際の値が異なる | → データフロー追跡（入力→変換→出力の各ポイント検査） |
| **SLOW** | パフォーマンス劣化 | → レイヤー別ボトルネック分離（NW / DB / CPU / レンダリング） |
| **INTERMITTENT** | 再現性が不安定 | → 共有状態・競合条件の洗い出し |
| **UI_BROKEN** | 表示・操作の不具合 | → コンポーネント状態検査 + レンダリングフロー追跡 |
| **INTEGRATION** | 外部サービス連携の不具合 | → コントラクト検証（リクエスト/レスポンス形状、認証、環境変数） |
| **NEW_FEATURE** | 新規機能の実装 | → 既存アーキテクチャの理解 + 類似機能のパターン調査 |
| **REFACTOR** | コードの構造改善 | → 依存グラフ解析 + 影響範囲の網羅的マッピング |

分類したら、該当する調査プロトコル（1-2）を選択する。
複数タイプに該当する場合は、**データ整合性に関わるもの**を最優先で調査する。

#### 再現条件の明確化

調査に入る前に、以下を整理する（情報がなければユーザーに確認）:
- **いつ**: いつから発生？特定の操作後？デプロイ後？
- **誰が**: 全ユーザー？特定ユーザー？特定の権限？
- **何をしたとき**: 再現手順（最小ステップ）
- **期待値 vs 実際**: 何が起きるべきで、何が起きたか
- **環境**: 本番？ステージング？ローカル？ブラウザ？

### 1-2. 仮説駆動調査（→ 調査エージェントに委譲）

**Explore サブエージェント**を**課題タイプに応じた専門指示**で起動する。
汎用的な「ファイルを探せ」ではなく、**タイプ別の調査プロトコル**を渡すこと。

#### 全タイプ共通: ベースライン調査

```
【全調査エージェント共通の指示】
1. キーワード Grep で関連ファイルを特定（コンポーネント名、関数名、テーブル名、APIエンドポイント）
2. 特定したファイルの import/export 関係から依存グラフを把握
3. 影響範囲を「直接変更が必要」「間接的に影響を受ける」「確認が必要」の3レベルに分類
4. DB マイグレーション、サーバーレス関数の関連箇所も確認
5. 型定義ファイルで関連する型の構造を把握
6. .claude/rules/review.md に定義されたプロジェクト固有ルールとの関連を確認
```

#### CRASH タイプ: コールスタック逆引きプロトコル

```
【CRASH 調査エージェントへの追加指示】
1. エラーメッセージの最後の行（実際のエラー）を起点にする
2. スタックトレースの最初のアプリケーションフレーム（自分たちのコード）を特定
3. そのファイルの該当行を Read で確認し、以下を報告:
   - エラーを引き起こす入力値は何か（null? undefined? 型不一致?）
   - その入力値はどこから来るか（props? API応答? ストア?）
4. 入力の供給元を1つずつ遡り、「最後に正しかったポイント」を特定
5. git log --oneline -20 -- {該当ファイル} で直近の変更を確認
6. 仮説を最低3つ、根拠とともに報告
```

#### WRONG_DATA タイプ: データフロー追跡プロトコル

```
【WRONG_DATA 調査エージェントへの追加指示】
1. データの流れを特定: ソース(DB/API) → 取得(hook/service) → 変換(util) → 表示(component)
2. 各ポイントでデータの形状（型・値）が期待通りか確認:
   - DB のテーブル定義・アクセス制御ポリシーを確認
   - fetch/query の WHERE 句・JOIN 条件を確認
   - 変換ロジック（map, filter, reduce）の正しさを確認
   - コンポーネントへの props の受け渡しを確認
3. 「最後に正しかったポイント」と「最初に不正になったポイント」を特定
4. 書き込み操作がある場合: トランザクション制御、部分失敗の可能性を確認
5. .claude/rules/review.md のプロジェクト固有チェック項目に関連する問題がないか確認
6. 仮説を最低3つ、根拠とともに報告
```

#### SLOW タイプ: パフォーマンス分離プロトコル

```
【SLOW 調査エージェントへの追加指示】
1. レイヤー別にボトルネック候補を分離:
   - DB: N+1 クエリ、欠落インデックス、重いJOIN、不要な全件取得
   - API: 不要な直列リクエスト、大きすぎるペイロード、キャッシュ未使用
   - フロント: 不要な再レンダリング、大きなバンドル、ブロッキング操作
2. DB 関連: マイグレーションファイルでインデックス定義を確認
3. API 関連: fetch/query の呼び出しパターンを確認（並列化可能か？）
4. フロント関連: メモ化（memo, useMemo, useCallback 等）の適用状況
5. git log --after="{劣化開始推定日}" --oneline で相関する変更を特定
6. ボトルネック候補を優先度順にリスト化して報告
```

#### INTERMITTENT タイプ: 競合条件調査プロトコル

```
【INTERMITTENT 調査エージェントへの追加指示】
1. 共有状態への全アクセスポイントを洗い出す:
   - グローバルストアへの書き込み箇所
   - 非同期 state の更新
   - リアルタイムサブスクリプション
   - ローカルストレージ / セッションストレージ
2. 競合条件の典型パターンを確認:
   - 非アトミックな read-modify-write
   - エフェクトのクリーンアップ漏れ
   - 非同期処理の完了順序の前提
   - Stale closure（古い変数キャプチャ）
3. 「この2つの操作が逆順で起きたらどうなる？」を各候補で検証
4. 仮説を最低3つ、再現条件の推定とともに報告
```

#### UI_BROKEN タイプ: コンポーネント状態調査プロトコル

```
【UI_BROKEN 調査エージェントへの追加指示】
1. 該当コンポーネントの状態管理を解析:
   - ローカル state の初期値と更新箇所
   - props の流れ（親コンポーネントからの受け渡し）
   - グローバルストア / Context からの購読
2. 条件付きレンダリングのすべてのパスを列挙
3. CSS/スタイリングの問題:
   - レスポンシブブレークポイントでの表示確認
   - z-index の競合、overflow の設定、position の問題
4. フレームワーク固有のルール違反の確認:
   - .claude/rules/review.md に定義されたプロジェクト固有チェック項目
   - フレームワークの既知のアンチパターン
5. 仮説を最低3つ、根拠とともに報告
```

#### INTEGRATION タイプ: コントラクト検証プロトコル

```
【INTEGRATION 調査エージェントへの追加指示】
1. 外部サービスとのコントラクトを検証:
   - リクエスト形状（ヘッダー、ボディ、認証情報）
   - レスポンス形状（期待する型 vs 実際の型）
   - エラーレスポンスの処理（HTTP 200 + エラーボディのケース含む）
2. 環境変数の確認:
   - ローカル / ステージング / 本番の設定差異
   - API キーの有効性、期限切れ
3. サーバーレス関数のコードパスを確認:
   - 認証チェックの有無
   - CORS 設定
   - タイムアウト設定
4. 連携先サービスのドキュメントとコードの整合性を確認
5. 仮説を最低3つ、根拠とともに報告
```

#### NEW_FEATURE / REFACTOR タイプ: アーキテクチャ理解プロトコル

```
【NEW_FEATURE / REFACTOR 調査エージェントへの追加指示】
1. 類似する既存機能のパターンを調査:
   - 同じドメインの既存実装パターン
   - ルーティング構造での配置パターン
   - 状態管理のパターン（グローバルストア or ローカル state or データフェッチライブラリ）
2. 影響範囲の網羅的マッピング:
   - 変更するモジュールを import している全ファイル
   - 共有する型定義の利用箇所
   - 関連するテストファイル
3. アーキテクチャ上の制約を確認:
   - DB スキーマの制約（FK, unique, check）
   - アクセス制御ポリシーの影響
   - 既存 API のバージョニング影響
4. パターンの一貫性を保つための推奨アプローチを報告
```

### 1-3. Git 考古学（→ 調査エージェントに委譲、1-2 と並列実行可）

**別の Explore サブエージェント**を起動し、Git 履歴から問題の時間的相関を調査させる。
1-2 の調査と**並列で実行**してよい。

```
【Git 考古学エージェントへの指示】

目的: 課題と変更履歴の相関を特定し、根本原因の候補を絞り込む

1. 直近の変更確認（バグ修正・パフォーマンス問題の場合）:
   git log --oneline -30
   → 課題発生時期と相関する変更を特定

2. ファイル単位の変更追跡（関連ファイルが特定されている場合）:
   git log --oneline -20 -- {ファイルパス}
   → 各関連ファイルの直近変更者・変更内容を確認

3. 特定行の変更追跡（疑わしい箇所がある場合）:
   git blame -L {開始行},{終了行} {ファイルパス}
   → いつ・誰が・なぜ変更したかを特定

4. 疑わしいコミットの diff 確認:
   git diff {コミットハッシュ}~1 {コミットハッシュ}
   → 具体的にどの行が変わったかを確認し、問題との因果関係を検証

5. キーワードによるコミット検索:
   git log --all --oneline --grep="{関連キーワード}"
   → 関連する過去の修正・議論を発見

6. 以下の形式で報告:
   - 相関の強い変更: [{コミットハッシュ}] {メッセージ} — {なぜ関連すると考えるか}
   - 相関の弱い変更: [{コミットハッシュ}] {メッセージ} — {念のため確認推奨}
   - 関連する過去の修正: [{コミットハッシュ}] {メッセージ} — {同じ領域の過去修正}
```

### 1-4. 根本原因の特定（メインエージェントが統合・判断）

調査エージェントたちの報告を受け取り、メインエージェントが**仮説を統合・検証**する。

#### ステップ1: 仮説の統合と優先順位付け

```
各調査エージェントが報告した仮説を統合し、以下の基準で優先順位を付ける:

🔴 最有力（複数の証拠が支持）:
  - 仮説: {内容}
  - 根拠: {調査エージェントA の発見} + {Git考古学エージェントの発見}
  - 検証方法: {最小コストの検証手段}

🟡 有力（単一の証拠が支持）:
  - 仮説: {内容}
  - 根拠: {発見}
  - 検証方法: {検証手段}

⚪ 可能性あり（消去法で残った候補）:
  - 仮説: {内容}
  - 検証方法: {検証手段}
```

#### ステップ2: 仮説の検証（必要な場合のみ）

最有力仮説が明確な場合はスキップして 1-5 へ進む。
不確実な場合は、**最小コストの検証**を追加の調査エージェントに委譲する:

```
検証方法の例:
- 特定の値をハードコードして挙動が変わるか確認（メンタル実行）
- 特定のクエリを直接 DB で実行して結果を確認
- 特定のコンポーネントの props を Read で確認
- 特定の条件分岐のすべてのパスを列挙して論理的に検証
```

#### ステップ3: 根本原因の確定

仮説が検証されたら、以下を明確にする:
- **根本原因**: 何が問題の本質か（1文で）
- **発生メカニズム**: なぜこの問題が起きるのか（因果の連鎖）
- **影響範囲**: この原因が影響する他の箇所はあるか
- **修正の方向性**: どう直せば根本的に解決するか

> **鉄則: 修正方針は根本原因に対して行う。症状を隠す修正は禁止。**
> 例: `null` でクラッシュ → `null チェック追加` ではなく `なぜ null が来るのか` を解決する

### 1-5. Issue 作成・ブランチ作成

#### ユーザー確認（AskUserQuestion）

Phase 1 の調査結果を以下の形式でユーザーに提示し、承認を得る:

```
## 調査結果サマリー

🔍 課題タイプ: {CRASH / WRONG_DATA / etc.}
🎯 根本原因: {1文で}
📊 発生メカニズム: {因果の連鎖}
📂 影響ファイル:
  - 直接変更: {ファイル一覧}
  - 間接影響: {ファイル一覧}
  - 確認必要: {ファイル一覧}

📋 Issue タイトル: {タイトル}
🔀 ブランチ名: {type}/#{仮番号}-{description}
✅ 受け入れ条件: {チェックリスト}

この分析で進めてよいですか？修正があれば教えてください。
```

ユーザーの承認後に Issue を作成し、ブランチを切る。

#### GitHub Issue 作成

```bash
gh issue create \
  --title "{課題タイトル}" \
  --body "$(cat <<'EOF'
## 課題
{課題の説明}

## 根本原因分析
{根本原因と発生メカニズム}

## 影響ファイル
{洗い出したファイル一覧（3レベル分類付き）}

## 受け入れ条件
{完了条件のチェックリスト}
EOF
)"
```

Issue番号を取得して以降のブランチ名・PR紐付けに使用する。

#### ブランチ作成

```bash
git checkout main
git pull origin main
git checkout -b {type}/#{issue番号}-{short-description}
```

ブランチ命名規則:
- `feat/`, `fix/`, `refactor/`, `docs/`, `test/`, `chore/`

## Phase 2: プラン策定・承認

### 2-1. 実装プランの作成（→ プランナーエージェントに委譲）

**Plan サブエージェント**を起動し、以下を含むプランを作成させる:

1. **ゴール定義**: 何を達成するか
2. **影響ファイル一覧**: 変更するファイルとその理由
3. **実装ステップ**: 具体的な手順（番号付き）
4. **技術的判断**: アーキテクチャ選択とその根拠

メインエージェントはプランの**要約を受け取り、ユーザーに提示**する。

### 2-2. 選択肢がある場合のユーザー確認

複数のアプローチが可能な場合、各選択肢を提示する:

```
### 選択肢A: {アプローチ名} ⭐（推奨）
- メリット: ...
- デメリット: ...
- 工数目安: ...

### 選択肢B: {アプローチ名}
- メリット: ...
- デメリット: ...
- 工数目安: ...
```

**⭐（推奨）を必ず1つ付ける。**

### 2-3. ユーザー承認 + 自動化レベル確認（AskUserQuestion）

プラン（単一 or 複数選択肢）を提示したら、**必ず AskUserQuestion で承認を取る**。
同時に、以降の進行モードを確認する:

```
上記のプランで実装を進めてよいですか？

📋 選択肢がある場合: どのアプローチを採用しますか？
✏️ 修正があれば教えてください

🚀 承認後の進め方:
  1. **通常モード** — 要所で確認しながら進めます（デフォルト）
  2. **全自動モード** — このままマージまで一気に進めます（確認なし、⭐推奨を自動採用）
```

- ユーザーが「2」「全自動」「おまかせ」「任せた」等と答えたら → 全自動モードに切り替え
- それ以外 → 通常モードで継続
- `--auto` フラグ付きで起動した場合はこの質問自体をスキップ

承認が得られるまで次のステップには進まない。

### 2-4. プランレビュー（2段階ゲート）⛔ MANDATORY — DO NOT SKIP

> **設計思想**: Claude Opus で反復修正（トークン安い）→ Codex CLI で最終ゲート（トークン高いが1回）。
> Codex のトークンを節約しつつ、外部視点の品質ゲートを確保する。

#### ステップ1: Claude Opus レビューループ（合格するまで繰り返す）

```
【ループ処理】
1. opus-code-review サブエージェント（または model: opus の general-purpose）を起動
   - レビュー対象: 実装プランのテキスト
   - 観点: 技術的妥当性、リスク、見落とし、プロジェクト固有ルール
2. P0/P1 の指摘があれば → プランを修正 → 再度 Claude Opus でレビュー
3. P0/P1 指摘なし（Claude Opus 合格）→ ステップ2 へ
```

#### ステップ2: Codex CLI 最終レビュー（1回のみ）

```
1. codex-code-review サブエージェントを起動（Bash で codex exec を実際に実行）
   - レビュー対象: Claude Opus 合格済みの実装プラン
   - 観点: Claude とは異なる外部LLMの視点での品質チェック
2. P0/P1 の指摘があれば → 修正 → ステップ1 に戻る（Claude Opus ループから再開）
3. P0/P1 指摘なし（Codex 合格）→ Phase 3 へ進む
```

## Phase 3: 実装

### 3-1. コーディング（→ 実装エージェントに委譲）

**general-purpose サブエージェント**を起動し、実装を委譲する:

```
サブエージェントへの指示:
- プロジェクトのコーディングルールに厳密に従うこと
- 各ファイル変更後にコミット（アトミックコミット）
- 完了したら変更ファイル一覧と変更概要を報告すること
```

**並列化**: 変更対象が独立している場合、複数の実装エージェントを並列起動する。
例: コンポーネント変更とサーバーレス関数変更を同時進行

メインエージェントは実装エージェントの完了報告を受け取り、Phase 3-2 へ進む。

### 3-2. 実装レビュー（2段階ゲート）⛔ MANDATORY — DO NOT SKIP

> **設計思想**: Phase 2-4 と同様、Claude Opus で反復修正 → Codex CLI で最終ゲート。

#### ステップ1: Claude Opus diff レビューループ（合格するまで繰り返す）

**opus-code-review サブエージェント（または model: opus の general-purpose）**を起動し、diff ベースの実装レビューを委譲する。
TypeScript の型チェック通過だけで完了にせず、**変更差分（diff）を基準に**実装レビューを行わせる。

```
【ループ処理】
1. git diff main...HEAD で変更差分を取得
2. 以下の5観点で差分を精査する:

   ✅ 仕様適合性:
     - 変更が Phase 2 で承認されたプランどおりか
     - Issue の受け入れ条件をすべて満たしているか

   ✅ ロジックの健全性:
     - 条件分岐・ループ・データフローに破綻がないか
     - 既存機能への副作用（意図しない挙動変化）がないか

   ✅ 変更の純度:
     - 不要な変更（フォーマット変更、無関係なリファクタ）が混ざっていないか
     - 変更の各行が課題解決に必要なものか

   ✅ エラーハンドリング・境界条件:
     - null/undefined、空配列、空文字列のケースは考慮されているか
     - API エラー、ネットワークエラー時の処理は適切か
     - ユーザー入力のバリデーションに漏れはないか

   ✅ プロジェクト固有ルール:
     - .claude/rules/review.md に定義されたチェック項目に違反していないか

3. P0/P1 の指摘があれば → 修正 → 再度 Claude Opus でレビュー
4. P0/P1 指摘なし（Claude Opus 合格）→ ステップ2 へ
```

#### ステップ2: Codex CLI 最終レビュー（1回のみ）⛔ MANDATORY — DO NOT SKIP

```
1. git diff main...HEAD で変更差分を取得
2. codex-code-review サブエージェントを起動（Bash で codex exec を実際に実行）
   - レビュー対象: git diff の実コード（Claude Opus 合格済み）
   - チェック項目: .claude/rules/review.md に定義されたプロジェクト固有チェック項目
3. P0/P1 の指摘があれば → 修正 → ステップ1 に戻る（Claude Opus ループから再開）
4. P0/P1 指摘なし（Codex 合格）→ Phase 4 へ進む
```

```
5. Claude Opus テスト実装レビューループ + Codex 最終レビュー

6. テスト実行

   - FAIL → 修正 → 再実行
```

## Phase 6: PR レビュー監視・対応

### 6-1. レビュー監視ループ

```
【監視ループ: 最大10分間、30秒間隔】

PR番号を取得後:

while (経過時間 < 10分):
    sleep 30秒

    reviews = gh api repos/{owner}/{repo}/pulls/{PR番号}/reviews
    comments = gh api repos/{owner}/{repo}/pulls/{PR番号}/comments

    if 新しいレビュー/コメントあり（bot以外）:
        → Phase 6-2 へ（レビュー対応）
        → 対応完了後、タイマーを0分からリセットして監視再開

監視終了（10分間レビューなし）→ Phase 6-3 へ
```

**監視対象**: 人間のレビュアーによるコメント
**無視**: CI/CD bot、デプロイ bot のコメント

### 6-2. レビュー対応サイクル

```
1. レビュー内容を確認
2. 修正プランを策定
3. Claude Opus レビューループ + Codex 最終レビュー（2段階ゲート）
4. 修正を実装
5. コミット & プッシュ
6. 監視タイマーを 0分 からリセットして監視再開
```

### 6-3. マージ

10分間新規レビューがなければマージ:

```bash
gh pr merge {PR番号} --squash --delete-branch
```

## レビューの共通ルール

### 2段階レビューゲートの設計思想

```
Claude Opus ループ（安い・反復向き）
    ↓ P0/P1 なしで合格
Codex CLI 最終ゲート（高い・外部視点・1回のみ）
    ↓ P0/P1 なしで合格 → 次の Phase へ
    ↓ P0/P1 あり → Claude Opus ループに戻る
```

- **Claude Opus**: 修正→再レビューの反復ループに最適（トークンコスト低い）
- **Codex CLI**: 異なるLLMの視点で最終チェック（Claude が見逃す問題を発見できる）

### 共通設定

- **合格基準**: P0/P1 の指摘が 0 件
- **P2（軽微）**: 報告のみ、修正は任意
- **最大リトライ**: Codex 5回不合格でユーザーに判断を仰ぐ
- **レビュー観点**: `.claude/rules/review.md` に準拠

## 中断・例外処理

- **ユーザーが中断指示**: 現在のステップを完了してから停止。進捗を報告
- **Codex レビュー5回不合格**: ユーザーに指摘内容を提示し、スキップか修正か判断を仰ぐ
- **テスト FAIL が解消できない**: 原因分析を報告し、ユーザーに判断を仰ぐ
- **ビルドエラー**: エラー内容を分析し、自動修正を試みる。3回失敗でユーザーに報告
